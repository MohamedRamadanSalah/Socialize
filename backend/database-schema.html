    POST_IMAGE {

    }

    LIKE {

    NOTIFICATION {

        uuid ActorId FK
        string Type "Like | Comment | Follow"
        uuid EntityId "nullable"
        bool IsRead
        timestamptz CreatedAt
    }

Composite-key tables (Like, Follow) have no surrogate Id — the pair itself is the key. Follow and Notification each carry two independent foreign keys into User.

03

Delete Behavior

not encoded in the diagram above — captured here instead

Mermaid’s ER notation doesn’t express ON DELETE rules, so every foreign key’s behavior is spelled out below. The pattern: children of a Post cascade with it; anything touching User restricts, since there’s no delete-account endpoint and several tables carry two independent foreign keys back to User.

RelationshipOn delete of parentRationale
Post → PostImageCascadeImages have no meaning without their post
Post → CommentCascadeComments have no meaning without their post
Post → LikeCascadeLikes have no meaning without their post
User → Post (author)RestrictNo delete-account endpoint exists; avoids surprise mass-deletes
User → Comment (author)RestrictSame reasoning; also dodges EF Core’s multiple-cascade-path error via Post
User → RefreshTokenCascadeTokens are meaningless without their owning user
User → Like (liker)RestrictAvoids a second cascade path into Like alongside Post → Like
User → Follow (either side)RestrictAvoids a cascade path colliding with the composite key’s dual FK
User → Notification (either side)RestrictSame reasoning as Follow
